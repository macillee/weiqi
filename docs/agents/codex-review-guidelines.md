# Codex Review 指南

> 项目：小棋童围棋闯关  
> 目标：让 Codex Review 聚焦产品边界、儿童体验、棋盘正确性、状态逻辑和代码质量。

---

# 1. Review 总原则

Codex Review 不只看代码能不能跑，还要检查是否符合本项目的产品目标。

本项目 v0.1 的核心是：

> 儿童 9 路围棋做题训练系统。

任何偏离这个核心、提前扩大范围的实现，都应被指出。

---

# 2. 产品边界检查

重点检查：

- 是否引入了 v0.1 不需要的复杂功能；
- 是否提前实现了后端、登录、支付、AI、在线对弈；
- 是否把 MVP 做成了复杂平台；
- 是否遵守 `docs/PROJECT_SPEC.md` 的范围定义。

应阻止：

- 为了“以后扩展”提前做复杂架构；
- 引入重型依赖解决简单问题；
- 用 AI/后端替代本地题库和本地进度。

---

# 3. 儿童体验检查

重点检查：

- 文案是否适合 6–10 岁儿童；
- 题目说明是否短而明确；
- 错误反馈是否温和；
- 页面是否过于复杂；
- 按钮是否足够大；
- 棋盘点击是否容易；
- 是否存在打击儿童积极性的表达。

推荐文案：

```text
差一点！先数一数白棋还有几口气。
```

不推荐文案：

```text
错误。
```

---

# 4. 棋盘正确性检查

重点检查：

- 坐标系统是否一致；
- x/y 是否都是 0-based；
- 棋子是否渲染在交叉点；
- 9 路棋盘是否有 9 条横线和 9 条竖线；
- 点击区域是否覆盖交叉点；
- 点击返回坐标是否准确；
- 棋盘边界是否正确；
- highlight 是否对应正确点位；
- disabled 状态是否生效。

---

# 5. 题目数据检查

重点检查：

- 题目 ID 是否唯一；
- boardSize 是否为 9；
- category 是否合法；
- level 是否在 1–5；
- initialStones 是否有重复坐标；
- answers 是否至少有一个；
- 所有坐标是否在棋盘范围内；
- hints 是否至少有一个；
- 文案是否儿童友好；
- 答案是否与题意一致。

---

# 6. 状态管理检查

重点检查：

- localStorage key 是否稳定；
- 进度刷新后是否不丢；
- attempts 是否正确记录；
- wrongProblems 状态是否正确流转；
- 星星奖励是否重复发放；
- mastered 错题是否从错题本消失；
- 每日练习是否避免重复题目过多；
- 使用提示是否被记录。

错题状态应为：

```text
active → reviewing → mastered
```

---

# 7. 推荐逻辑检查

每日练习推荐规则：

```text
40% 错题复习
40% 当前等级题
20% 挑战题
```

重点检查：

- 错题不足时是否能补题；
- 题目不足时是否能 gracefully fallback；
- 是否会出现空练习；
- 是否会因为数据不足导致页面崩溃。

---

# 8. 代码质量检查

重点检查：

- TypeScript 类型是否清晰；
- 组件职责是否单一；
- GoBoard 是否只负责棋盘展示和点击；
- ProblemPlayer 是否只负责题目流程；
- 业务逻辑是否集中在 lib 中；
- 是否存在重复逻辑；
- 是否存在过度耦合；
- 是否有核心测试；
- 是否有无用依赖；
- 是否有 console error；
- 是否有明显性能问题。

---

# 9. UI 与移动端检查

重点检查：

- 移动端棋盘是否能正常显示；
- 点击区域是否足够大；
- 页面是否有横向滚动；
- 字号是否适合儿童；
- 反馈弹窗是否遮挡核心信息；
- 颜色是否清晰；
- 动画是否不过度。

---

# 10. PR Review 输出建议

Codex Review 可以按以下格式输出：

```md
## Review Summary

本 PR 完成了什么，整体是否符合 v0.1 范围。

## Blocking Issues

必须修复的问题。

## Non-blocking Suggestions

可以后续优化的问题。

## Product Fit

是否符合儿童围棋做题训练定位。

## Test Notes

建议补充或确认的测试点。
```

---

# 11. 必须要求修改的情况

出现以下情况应 request changes：

- 引入 v0.1 明确禁止的复杂功能；
- 棋盘点击坐标错误；
- 答案判断错误；
- 题目数据坐标越界；
- 错题本无法工作；
- localStorage 进度刷新丢失；
- TypeScript 编译失败；
- lint 失败；
- 儿童文案明显不合适；
- 页面核心流程无法完成。

---

# 12. 可以接受但建议后续优化的情况

以下情况可 comment，不一定阻塞：

- UI 不够精美；
- 动画较少；
- 题库数量未完全补齐但架构正确；
- 报告指标较简单；
- 部分边缘状态提示不够细；
- 测试覆盖率不高但核心流程可手测。
