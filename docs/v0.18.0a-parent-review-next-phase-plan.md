# v0.18.0a Parent Review Next Phase Plan

> Planning document covering goal/constraints, v0.17 assets, 4 candidate integration surfaces, session boundary plan, data flow, privacy/safety review, UI/UX principles, slice plan (a–d), and v0.18.0b acceptance criteria.

## Goal and Constraints

### Goal

Develop a parent review integration surface that works seamlessly with the existing v0.17 parent review debug surface prototype, while addressing the limitations and providing a production-ready solution for actual parents (not just developers).

### Constraints

1. **Privacy-First**: Must maintain strict privacy boundaries and data minimization
2. **Child-Facing Focus**: Interface should be warm, concrete, and age-appropriate (6-10 years)
3. **Local-First**: Should not break existing local anonymous mode
4. **Backward Compatible**: Must work alongside existing v0.17 debug surface
5. **Lightweight**: Should be efficient and not impact existing app performance
6. **Auth-Independent**: Should work with both authenticated and unauthenticated users

### v0.17 Assets to Build Upon

- `/dev/session-summary` - Parent session summary debug page
- `src/lib/session-summary-input.ts` - Mapping helper for session summary
- `ParentSessionSummary` component - Full rendering with category/level tables
- `src/lib/session-summary-helper.ts` - Pure local summary aggregation
- 17 unit tests for session summary functionality

## Candidate Integration Surfaces

Based on v0.17.0a analysis, we recommend **Surface B: Parent-Facing Learning Report Integration** as the primary target for v0.18.0a.

### Surface A: Existing Debug Panel
- **Pros**: Fast to implement, leverages existing v0.17 prototype
- **Cons**: Developer-focused, not parent-friendly, requires auth
- **Status**: v0.17.0b delivered

### Surface B: Parent-Facing Learning Report Integration ⭐ (Recommended)
- **Location**: Main `Report` page with parent-specific insights
- **Integration**: Smooth transition from `/practice` to `/report` flow
- **Benefits**: Natural parent journey, aligns with existing app structure
- **Target Users**: Authenticated parents viewing child's progress

### Surface C: Floating Parent Dashboard Widget
- **Location**: Home page or practice completion overlay
- **Pros**: Minimal friction, visible at key learning moments
- **Cons**: Less comprehensive, harder to navigate complex data

### Surface D: Separate Parent Portal Page
- **Location**: `/parent` or dedicated section
- **Pros**: Comprehensive, can include multiple view modes
- **Cons**: Higher friction, breaks current app flow

## Session Boundary Plan

### Data Flow

```
Child Practices → localStorage (Progress, Attempts) 
                    ↓
Parent Session Summary Helper → ParentSessionSummary (aggregated, sanitized)
                    ↓
Debug Surface (dev) vs Parent Surface (prod)
                    ↓
Parent Learning Report (UI)
```

### Privacy Boundary

1. **Raw Data**: All raw attempt/answer data stays in child's localStorage
2. **Aggregated Summary**: `ParentSessionSummary` contains only aggregated insights
3. **No Raw Answers**: Parent surface never shows child's specific answers
4. **No Child Identification**: Parent surface only shows progress patterns

### Data Minimization

- Only includes: problem categories, accuracy rates, time patterns
- Excludes: specific correct/incorrect answers, timing details
- Aggregates to: category-level and session-level insights

## UI/UX Principles

### Child-Friendly Design

- **Warm, Concrete Language**: "宝宝今天学得真好" instead of "High performance"
- **Visual Consistency**: Matches existing app design language
- **Clear Actions**: Simple, actionable buttons and navigation
- **No Technical Jargon**: Avoids terms like "分析", "统计", "数据"

### Information Architecture

- **Above the Fold**: Key insights first (今天完成, 本周表现)
- **Progressive Disclosure**: Details available on demand
- **Mobile-Responsive**: Works on tablets and phones
- **Accessibility**: Screen reader friendly, sufficient contrast

### Interaction Flow

1. **Parent Login** (if applicable) → **Child Selection**
2. **Session Summary View** → **Detailed Breakdown**
3. **Suggestions** → **Action Items**
4. **Privacy Notice** → **Data Control**

## Acceptance Criteria

### Functional Requirements

1. **Parent Access**: Authenticated parents can view child progress summaries
2. **Privacy Compliance**: No exposure of child's raw answer data
3. **Mobile Compatibility**: Works on phones, tablets, and desktop
4. **Performance**: Loads in <2 seconds with minimal data transfer
5. **Backward Compatibility**: Does not break existing local anonymous mode

### Quality Requirements

1. **User Experience**: Warm, non-judgmental language and tone
2. **Accessibility**: WCAG 2.1 AA compliance
3. **Test Coverage**: 80%+ test coverage for new components
4. **Documentation**: Clear, child-friendly documentation
5. **Error Handling**: Graceful degradation when features fail

### Privacy Requirements

1. **Data Minimization**: Only aggregated insights stored and transmitted
2. **No PII**: Parent authentication not required for access
3. **Local Storage**: All sensitive data remains in localStorage
4. **No Tracking**: No analytics or user tracking in parent view

## Risks and Mitigation
